Two-password model shipped 2026-05-21 (migration 0011). Audited, verdict: ship-able, no 🔴/🟠 blockers. Backs REQUIREMENTS AC-002.1..AC-002b.5.

**Field map (the separation that must never collapse):**
- `users.login_password_hash` (schema.ts:65) = Argon2id of the LOGIN password. ONLY `POST /auth/login` (auth.ts:126) verifies it. NULL = SSO-only/legacy → cannot email+password login (dummy-verify at auth.ts:96 keeps timing flat, no enumeration).
- `users.password_hash` (schema.ts:60) = Argon2id of the MASTER password. ONLY consulted by `/me/verify-password` (me.ts:538), `/me/password/setup` (me.ts:322/352), recovery-kit regenerate/reset, and the recovery flow. NEVER by login. Drives `requiresPasswordSetup = passwordHash===null` (me.ts:187).
- `requireVaultUnlocked` (middleware/auth.ts:108) gates plaintext-returning routes (items GET, sends POST, attachments download) on `sessions.vault_unlocked_at` freshness only.

**Register (`POST /auth/register`, auth.ts:241):** sets login_password_hash ONLY, master NULL, org-less (no membership), NO recovery kit (kit is master-bound). 5/hr/IP RL. email_taken pre-check + unique-index race catch (isUniqueViolation→emailTaken, auth.ts:297). Mints session immediately. Strength min(10) == setup bar. errors.emailTaken = 409 email_taken (errors.ts:51).

**Verified invariants (authRegister.test.ts, 5/5 pass, real PG):** master alone does NOT log in (401); login pw does (200); SSO-only/null-login-hash can't log in (401); register leaves master NULL + requiresPasswordSetup true + org-less.

**Invite signup-and-accept aligned to 2-pw model (audited 2026-05-21, focused round — SHIPPABLE, no 🔴/🟠):** `POST /invite/:token/signup-and-accept` (invitations.ts:341) now mirrors /auth/register exactly: sets `loginPasswordHash` only (invitations.ts:448), master `passwordHash:null` (449), NO recovery kit emitted here, response carries NO recoveryCode (the prior bug set master+no-login-hash → permanent lockout). Recovery kit minted only at /me/password/setup (me.ts:334-335, atomic conditional UPDATE guarded by isNull(passwordHash)). emailVerifiedAt=now justified (invite token proves mailbox ownership). Same Argon2 `hashPassword`, same strength floor min(10), same createSession+cookie path as register/login. Invite integrity NOT regressed: email-match is the SIGNUP gate via row.email→users.email insert (no session yet, so squatted-account hijack N/A — squatter would need the token AND can't pre-create the row, else 409 user_exists at invitations.ts:411), role copied from `row.role` server-side (not client), token validation revoked/accepted/expired/user_exists all retained, single tx (user+membership+accept-flip+audit) at invitations.ts:435. Audit action stays `member.invitation_accepted` (metadata.viaSignup:true); `recovery_kit_generated` correctly removed here (none minted). RL 5/min/IP+token. Tests inviteSignup.test.ts 4/4 pass real PG (login-after-signup proves no lockout). FE invite/[token]/page.tsx signup branch: no recoveryCode/RecoveryKitModal, explicitly does NOT stamp unlock (vault stays locked until master set — page.tsx:242-248), refreshAuth→/app→SessionGuard routes requiresPasswordSetup→/setup-password (session-guard.tsx:63-64). Residual: same vault_unlock auto-stamp at createSession (below) — bounded: invite-signup user is master-NULL so no plaintext readable regardless.

**KEY RESIDUAL (WARN-class, NOT a blocker — pre-existing WARN-I interaction newly relevant):** `createSession` (session.ts:69) stamps `vaultUnlockedAt=now` on EVERY new session. Pre-split, a fresh login proved the MASTER pw (login==master), so the auto-unlock was justified. Post-split, login proves only the LOGIN pw, so the WARN-I server-side vault gate is now satisfied by login-pw possession alone for the first 15 min after login — master pw no longer required to read secrets in that window. Real impact bounded: a register-only user (master NULL) is org-less so loadItemForUser→loadVaultForUser returns nothing (no plaintext). For a master-set user it's a genuine narrowing of the Phase A.5 boundary. Phase A.5 hardening recommendation: do NOT auto-stamp vault_unlocked_at at login session creation for users who have a master set; require an explicit /me/verify-password. Phase C makes moot.

**Email-verification residual (accepted this round):** no email verification at register → squatting `victim@corp.com`. Invite-accept is safe: requires session-email === invitation-email (invitations.ts:213, 403 invitation_email_mismatch) AND token. SSO domain-bypass: `/auth/register` has NO domain allowlist, but SSO `GOOGLE_OAUTH_ALLOWED_DOMAIN` (sso.ts:160/255) still gates Google. A user can register email+password with a corp-SSO domain — only a policy/onboarding concern, not an auth bypass (they get an org-less account, no workspace access without an invite that re-checks email).

**LOGIN PASSWORD HAS NO RESET/CHANGE PATH (🟡 availability gap, surfaced by the sweep — recheck each audit):** `loginPasswordHash` is written ONLY at register (auth.ts:274) and invite-signup (invitations.ts:427). There is NO endpoint to reset or change it. The only "Forgot password?" flow is `/forgot-password` → `/auth/password/reset-with-recovery` (auth.ts) which sets `passwordHash` (the MASTER), recoveryKitHash=null, deletes all sessions — it does NOT touch loginPasswordHash. So a non-SSO user who forgets their LOGIN password is permanently locked out; the login page's "Forgot password?" link (login/password/page.tsx:279→/forgot-password) routes them to a master-reset that won't help. Not a secret leak / auth bypass (so not a ship blocker), but a real recoverability gap. Phase B candidate: a login-password reset (email magic-link) or fold login-pw reset into the recovery flow.

**FALSE zero-knowledge copy in Phase A (🟡 pre-existing, NOT at sign-in surfaces — flagged under focus "retained ZK messaging accuracy"):** Phase A is backend-held KEK (no client-side crypto), so any "encrypted in your browser" claim is inaccurate. Live strings: `ni.security_hint` (translations.ts:1709-1711, rendered new-item-dialog.tsx:683) "All fields are encrypted in your browser with AES-256-GCM before being sent to the server"; `recip.encryption_value` (translations.ts:1979) "AES-256-GCM zero-knowledge" (one-time-send recipient view). The sign-in sweep correctly scrubbed login-page ZK claims; these two are in vault/send UIs and predate the sweep. Re-flag until Phase C or until copy is softened to "encrypted at rest with AES-256-GCM".

**Redact:** req.body.password + *.password cover login/register bodies (logger.ts:12/19). displayName not secret. login_password_hash is a DB column, never in a log path.
